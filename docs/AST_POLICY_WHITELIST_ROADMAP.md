# AST Policy Whitelist Roadmap

## 1. Purpose

Cyber-Immunizer candidate detectors are generated by an LLM and must be treated
as untrusted code. Before any candidate detector is evaluated or promoted, it is
validated by `core/policy.py` against a set of AST safety checks.

The current policy is primarily **blacklist**-oriented: it enumerates constructs
that are forbidden (dangerous builtins, forbidden modules, dunder access, and
now structurally unnecessary control-flow constructs) and rejects any candidate
that contains them. This is a safe starting point, but a blacklist-only policy
has long-term limitations:

- A blacklist can be bypassed if an attacker finds a construct not yet enumerated.
- As the language evolves, new bypass paths may be introduced.
- It is difficult to verify that a blacklist is complete.

A future **whitelist** model — where only explicitly permitted constructs are
accepted and everything else is rejected by default — is a stronger defense. This
document defines a staged migration plan toward that goal.

This is a Phase 2.5 Stage A document. The allowlist migration is not yet
complete. Later stages are described in Section 6.

## 2. Current Phase 2.5 Policy

### 2a. Existing Blacklist Checks

`core/policy.py` enforces the following checks as of Phase 2.5:

**Forbidden builtins:**
`eval`, `exec`, `compile`, `__import__`, `open`, `globals`, `locals`, `vars`,
`input`, `getattr`, `setattr`, `delattr`, `type`, `dir`, `super`, `breakpoint`,
`callable`

**Forbidden modules:**
`os`, `subprocess`, `socket`, `pathlib`, `shutil`, `requests`, `urllib`, `sys`

**Only allowed import:**
`from core.types import ...`

**Dunder attribute rejection:**
Any `ast.Attribute` node where the attribute name starts and ends with `"__"`.
This blocks `request.__class__`, `request.__dict__`, `inspect_request.__globals__`,
and any other dunder attribute access.

### 2b. Complexity and Resource Guards

- Source size limit (`MAX_POLICY_SOURCE_CHARS`)
- AST node count limit (`MAX_AST_NODES`)
- AST nesting depth limit (`MAX_AST_DEPTH`)
- String/bytes literal size limit (`MAX_LITERAL_CHARS`)
- Collection literal size limit (`MAX_COLLECTION_ITEMS`)
- Range iteration count limit (`MAX_RANGE_CONSTANT`)
- Sequence repeat multiplier limit (`MAX_REPEAT_MULTIPLIER`)
- List/set/dict comprehension rejection
- Standalone generator expression rejection
- Non-constant range() argument rejection
- Non-constant repeat multiplier rejection
- `join(generator)` restricted to `request.query.items()` / `request.headers.items()` only

### 2c. New Structural Disallowed Node Guard (This PR — Stage A)

Phase 2.5 adds `check_disallowed_ast_constructs()` which rejects the following
constructs that are not needed for baseline-compatible detection logic. Violation
messages use the stable phrase `"disallowed AST construct"`.

Rejected node types:

- `try` / `except` (`ast.Try`, `ast.ExceptHandler`)
- `raise` (`ast.Raise`)
- `with` / `async with` (`ast.With`, `ast.AsyncWith`)
- `lambda` (`ast.Lambda`)
- `while` (`ast.While`)
- `yield` / `yield from` (`ast.Yield`, `ast.YieldFrom`)
- `await` (`ast.Await`)
- `global` (`ast.Global`)
- `nonlocal` (`ast.Nonlocal`)
- `del` (`ast.Delete`)
- walrus operator `:=` (`ast.NamedExpr`)
- `assert` (`ast.Assert`) — can raise `AssertionError` and is silently stripped under `python -O`
- `match` / `case` (`ast.Match`, Python 3.10+)
- `async def` (`ast.AsyncFunctionDef`)
- `class` (`ast.ClassDef`)
- Nested or extra `def`: any `ast.FunctionDef` other than the single
  top-level `inspect_request` function is rejected. This covers nested
  helpers, duplicate definitions, and top-level helper functions.

### 2d. Indirect Dunder-Access Risk

Phase 2.5 rejects direct dunder attribute access through `ast.Attribute`
inspection via `check_dunder_access()`. This blocks direct forms such as:

    request.__class__
    request.__dict__
    inspect_request.__globals__

This is necessary but not a mathematical proof that every indirect path to
dunder attributes is impossible. Later allowlist stages must review whether
the remaining permitted call and attribute patterns can create indirect
access to Python internals. Until then, major reflection helpers remain
forbidden and direct dunder attributes remain rejected.

Current policy also blocks major reflection helpers — `getattr`, `setattr`,
`delattr`, `globals`, `locals`, `vars`, `dir`, `type`, `super`,
`__import__` — which eliminates the most common indirect access vectors.

Phase 2.5 rejects direct dunder attribute access through `ast.Attribute`
inspection. This is necessary but not a mathematical proof that every
indirect path to dunder attributes is impossible. Later allowlist stages
must evaluate whether indirect access vectors remain possible through
allowed call and attribute patterns. This PR does not fully solve the
indirect dunder-access problem.

## 3. Future Whitelist Target

The long-term goal is to replace the blacklist with an explicit allowlist:
a policy that accepts only constructs that have been explicitly reviewed and
permitted. Everything else is rejected by default.

The staged allowlist migration is ongoing. A full migration requires:

- Auditing every AST node type that appears in valid candidate detectors.
- Defining explicit per-category rules for calls, attribute access, and operators.
- Ensuring the baseline `core/detector.py` passes the stricter policy.
- Adding regression tests for every permitted pattern.

This PR is Stage A only. The migration to a complete allowlist policy
remains future work.

## 4. Allowed Baseline-Compatible Categories

The following constructs are present in the baseline `core/detector.py` and are
expected to remain permitted in future allowlist stages:

- Module docstring (a single leading string constant at module level)
- `from core.types import Request, DetectionResult`
- Top-level function: `def inspect_request(request: Request) -> DetectionResult:`
- Assignments to local variables (`x = ...`)
- `if` statements over local conditions
- `for` loops over bounded local token collections
- Attribute reads: `request.method`, `request.path`, `request.query`,
  `request.headers`, `request.body`
- `.lower()` string method
- `.items()` only on `request.query` and `request.headers`
- `" ".join(...)` over a bounded `request.query.items()` /
  `request.headers.items()` generator expression
- `list.append(...)` to a local `matched` list
- Built-in calls: `len`, `min`, `tuple`
- `DetectionResult(...)` construction
- Integer and string constants, simple tuples, lists, simple arithmetic for
  confidence scoring
- f-strings for safe reason text (e.g., `f"matched {token!r}"`)

## 5. Explicitly Disallowed Constructs

The following constructs are explicitly disallowed because they are not needed
for baseline-compatible detection logic and they represent either unnecessary
complexity or a meaningful attack surface:

| Construct | Reason |
|-----------|--------|
| `try` / `except` | Not needed; exception handling in candidates increases complexity |
| `raise` | Not needed; candidates must return `DetectionResult`, not raise |
| `with` / `async with` | Not needed; context managers are not part of the detection model |
| Nested or extra `def` | Not needed; nested helpers increase bypass surface |
| `class` | Not needed; no class definitions are required |
| `lambda` | Not needed; lambda is a nested function in disguise |
| `while` | Not needed; only bounded `for` loops over local collections are needed |
| `yield` / `yield from` | Not needed; candidates must not be generators |
| `await` / `async def` | Not needed; candidates are synchronous |
| `global` / `nonlocal` | Not needed; candidates must not modify enclosing scope |
| `del` | Not needed |
| `assert` | Not needed; can raise `AssertionError` instead of returning `DetectionResult`; behavior changes silently under `python -O` |
| `match` / `case` | Not needed |
| walrus operator `:=` | Not needed |
| Any import except `from core.types import ...` | Not needed |
| Reflection / dynamic execution | Blocked: `eval`, `exec`, `getattr`, etc. |
| File / network / process operations | Blocked: `os`, `subprocess`, `socket`, etc. |

## 6. Migration Stages

### Stage A — Document roadmap, reject obviously unnecessary constructs (this PR)

- Add this document.
- Add `check_disallowed_ast_constructs()` to reject the constructs listed in
  Section 5.
- Add regression tests proving baseline-safe patterns pass and high-risk
  constructs are rejected.

### Stage B — Add explicit call/attribute allowlist tests

- Define an explicit list of permitted method calls (e.g., `.lower()`,
  `.items()`, `.append()`).
- Define an explicit list of permitted attribute accesses (e.g.,
  `request.method`, `request.path`).
- Add tests asserting that non-permitted calls and attributes are rejected.
- Evaluate whether remaining indirect dunder-access vectors are possible through
  the allowed call/attribute patterns identified in this stage.

### Stage C — Implement mutation-region-only allowlist

- Apply the allowlist only to the mutation region (between `MUTATION_START`
  and `MUTATION_END`).
- Allow the stable boilerplate outside the mutation markers to use a broader
  set of constructs.

### Stage D — Require all new policy exceptions to include tests

- Any future change that permits a previously-disallowed construct must include
  a test demonstrating that the construct is safe in the intended context.

## 7. Non-Goals

The following are explicitly out of scope for this PR and for Phase 2.5:

- **Do not connect the Gemini API.** Phase 3 activation is not part of this PR.
- **Do not activate Phase 3.** `live_model_enabled` remains `false`. Phase 3 activation
  requires separate review and approval.
- **Do not relax existing policy.** No existing blacklist check is weakened.
- **Do not introduce real exploit payloads into tests.** All test indicators use
  neutralized symbolic names.
- **Do not claim the staged allowlist migration is finished.** This PR is Stage A only.
  Sections 3 and 6 describe remaining future work.
